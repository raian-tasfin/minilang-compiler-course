Given those two lookup structures (`symbol_at_register[reg] -> symbol`, `register_of_symbol[symbol_id] -> reg`), here's the allocate/free logic per unit `u`, run in iteration order over `def[u]` and `in[u]`/`out[u]`:

```
// ALLOCATE: every symbol defined in this unit needs a register
for s in def[u]:
    if register_of_symbol[s.id] == NONE:
        reg = find_free_register()   // scan symbol_at_register for an empty slot
                                      // (or pull from a free-list if you keep one)
        if reg == NONE:
            spill_or_error()         // no free register available
        register_of_symbol[s.id] = reg
        symbol_at_register[reg] = s

// FREE: every symbol alive entering but dead leaving just had its last use
for s in in[u]:
    if s not in out[u]:
        reg = register_of_symbol[s.id]
        register_of_symbol[s.id] = NONE
        symbol_at_register[reg] = NONE
```

Order matters: run the **allocate** pass before emitting the unit's instruction (you need the register in hand to generate the actual operand), and run the **free** pass after emitting it (the symbol is still needed as an operand during this unit even if it dies here). So the real sequence per unit is:

```
allocate_pass(u)         // for s in def[u]
emit_instruction(u)      // uses register_of_symbol[...] for operands
free_pass(u)              // for s in in[u] - out[u]
```

One subtlety worth flagging: for a binop like `dest = val1 + val2`, `dest ∈ def[u]` gets allocated, but `val1`/`val2` are only in `use[u]`, not `def[u]` — they should already have registers from when *they* were defined earlier, so the allocate pass correctly skips them (the `register_of_symbol[s.id] == NONE` check is what makes this safe — already-allocated symbols are no-ops in that loop, it's really only acting on genuinely new symbols).
